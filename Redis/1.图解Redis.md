## 什么是Redis？

redis是一个基于内存的数据库，读写速度快，经常用来做缓存



## 什么是AOF日志（Append Only File）？

AOF日志记录了所有写操作命令，作用是实现数据的持久化，在redis宕机重启时，通过重新执行命令来恢复数据

![image-20260314215314614](../assets/1.图解Redis/image-20260314215314614.png)

- 写入时机，操作执行成功后，才记录AOF日志
- 写入时机好处，① 保证AOF日志中的命令都是正确的，避免了检查命令语法的开销；② 不阻塞当前操作
- 风险，① 操作执行成功后，AOF日志没来得及记录，造成数据丢失；② 主进程记录AOF日志，如果IO压力过大，可能阻塞下一个操作



## 写AOF日志过程？

redis执行写操作后，将命令先写入缓冲区，根据不同的写回策略，决定写入磁盘的时机

![image-20260314221102098](../assets/1.图解Redis/image-20260314221102098.png)

- redis执行写操作后，将命令写入aof_buf缓冲区
- 调用write()，将aof_buf缓冲区数据拷贝到page cache内核缓冲区
- 依据不同的回写策略，决定写入磁盘时机



## AOF写回策略？落盘策略？

redis写回策略由appendfsync参数控制，有三种策略，本质上是控制fsync()函数调用时机

- always，执行操作后，【主进程】将命令写入磁盘

- everysec（默认），执行操作后，【主进程】将命令写入缓冲区，【后台线程】每隔1秒将缓冲区数据写入磁盘

- no，执行操作后，【主进程】将命令写入缓冲区，由【操作系统】决定什么时候将缓冲区数据写入磁盘（30s左右）

- 查看配置（默认everysec）

  ```redis
  config get appendfsync
  ```



## 什么是重写AOF？

AOF重写，是将每个键值对用一条命令记录到新的AOF日志中，作用是实现AOF日志的压缩，防止AOF日志无限膨胀



## 重写AOF，使用子进程的好处？

重写AOF需要扫描所有键值对并写入，是重载操作，使用子进程有2个好处

- 不会阻塞主进程，重写需要扫描所有键值对，时间长，交给子进程完成不会阻塞主进程，主进程可以继续执行命令
- 子进程和父进程共享内存数据（以只读的方式共享），如果使用多线程，在修改共享内存数据时，需要通过加锁保证数据安全，影响性能



## 重写AOF，父子进程如何进行内存数据共享？

主进程通过fork创建子进程时，主进程的把【页表】复制给子进程，页表记录了虚拟地址和物理地址的映射，2个不同页表映射同一个物理地址，实现父子进程共享内存数据



## 重写AOF，什么情况下子进程会阻塞父进程？

2个阶段

- 创建子进程时，需要复制父进程页表给子进程，页表越大，阻塞时间越长
- 触发写时复制，父进程修改数据时，会触发写时复制，需要复制一份内存数据给子进程，修改的数据越多，需要复制的物理内存越大，阻塞时间越长

## 什么是DRB快照（Redis Database Backup）？

DRB快照是全量数据的快照

- 执行save命令，在主线程生成DRB文件

- 执行bgsave命令，创建子线程生成DRB文件

- 优点，恢复速度快，恢复时，直接将DRB文件读进内存

- 缺点，生成DRB快照开销大，如果生产频率过高，性能开销大，如果生成频率过低，丢失数据多

- 查看配置（默认无）

  ```
  config get save
  ```



## 生成DRB快照时，数据能被修改吗？

可以，拥有写时复制（COW，Copy On Write）技术



## 什么是写时复制？

创建子进程后，子进程和父进程共享同一个内存页，当父进程尝试修改数据时，才复制一份新的内存页给子进程，作用是节省内存，降低主进程阻塞时间

- fork子进程时，复制父进程页表给子进程，父子进程共享同一个内存页

- 父进程尝试修改数据时，复制一份新内存页给子进程



## 写时复制的风险？

/

- 创建子进程时，需要复制父进程页表给子进程，页表越大，主进程阻塞时间越长
- 父进程修改数据时，会触发写时复制，需要复制一份内存数据给子进程，修改的数据越多，需要复制的物理内存越大，可能会造成内存翻倍，甚至内存溢出



## 什么是混合持久化（AOF和DRB合体）？

混合使用AOF和DRB，在重写AOF日志时，先将全量数据用DRB格式写入AOF文件，再将增量命令用AOF格式写入AOF文件，作用是恢复速度快，丢失数据少	



# 功能篇

## key如何设置过期时间？

执行命令

```
set key value px 1000 # 设置1000ms后过期
pexpire key 2000 # 设置2000ms后过期
ttl key # 数字代表过期剩余时间，-1代表不过期，-2代表已过期
```



## redis如何判定key已过期？

对key设置过期时间时，redis会把key放入过期字典，查询时，检查key在不在过期字典中，并且判断过期时间是不是在系统时间之前

- 过期字典数据结构，哈希表，key是指针，指向某个键对象，value是long long类型整数，记录key过期时间
- 过程1，对key设置过期时间时，redis把key放入过期字典，
- 过程2，查询key时，redis检查key在不在过期字典中，如果不在，返回键值，如果在，和系统时间比对，如果比系统时间小（在系统时间之前），判定key已过期



## 过期删除策略有哪些？

定时删除

- 做法，在设置key过期时间时，创建一个定时事件，当到达过期时间时，执行删除key操作
- 优点，保证key会被马上删除
- 缺点，过期key较多时，对CPU消耗大

惰性删除

- 做法，不主动删除过期key，每次数据库访问key时，检查key是否过期，如果过期则删除
- 优点，对CPU消耗小
- 缺点，如果key一直不被访问，key无法被删除，造成内存浪费

定期删除

- 做法，定期随机从数据库取出一定数量的key进行检查，并删除过期key
- 优点，对CPU消耗可控，和惰性删除相比，能删除较多的过期key
- 缺点，难以确定删除操作的频率，如果太频繁，和定时删除一样，对CPU消耗大；如果频率太小，和惰性删除一样，造成内存浪费



## redis过期删除策略是什么？

惰性删除 + 定期删除

- 惰性删除，访问key时，检查key是否过期，如果过期则删除
- 定期删除，定期【随机】从数据库取出一定数量的key进行检查，并删除过期key，默认每秒10次，每次抽取20个key，如果过期key数量占比大于25%，重复定期删除步骤，重复过程超过25ms结束



## 什么是内存淘汰策略？

内存超过上限值后，使用内存淘汰策略，删除符合条件的key

- 查看配置

  ```
  CONFIG GET maxmemory
  ```



## redis内存淘汰策略有哪些？

redis内存淘汰策略

【不进行数据淘汰】

- noeviction（默认），超过最大内存时，不淘汰数据。写入报错，读取成功

【在设置了过期时间的键值中，进行淘汰】

- volatile-random，在设置了过期时间的键值中，随机淘汰
- volatile-ttl，在设置了过期时间的键值中，淘汰最早过期的键值
- volatile-lru，在设置了过期时间的键值中，淘汰最久未使用的键值
- volatile-lfu，在设置了过期时间的键值中，淘汰最少使用的键值

【在所有键值中，进行淘汰】

- allkeys-random，在所有键值中，随机淘汰
- allkeys-lru，在所有键值中，淘汰最久未使用的键值
- allkeys-lfu，在所有键值中，淘汰最少使用的键值



## 什么是传统LRU（Least Recently Used）？

淘汰最久未使用的键值

- 做法，基于链表实现，节点被访问时，移动节点到链头，链尾节点是最久未使用的节点，进行内存淘汰时，删除链尾节点



## 什么是redis的近似LRU？

redis的LRU和传统LRU不同，进行内存淘汰时，随机挑5个key，淘汰最久未使用键值

- 做法，在redis对象中，加入字段lru表示【最后一次访问时间】，进行内存淘汰时，随机挑5个key，淘汰最久未使用键值
- 好处1，空间上，不用维护链表，节省空间
- 好处2，时间上，数据被访问时，不用将数据移到链表头，节省时间
- 缺点，缓存污染，只读取一次的键值，要存活很久才被删除



## 什么是传统LFU（Least Frequently Used）？

淘汰最少使用的键值

- 做法，LFU记录了每个数据的访问次数，进行内存淘汰时，淘汰最少使用的键值



## 什么是redis的近似LFU？

在redis对象中，复用字段lru表示【访问时间戳 + 频率】，访问键值时，对频率做衰减和增加操作，淘汰最少使用的键值

- 做法，在redis对象中，复用字段lru，前16位表示访问时间戳，后8位是访问频率
- 频率变化，频率默认是5，键值被访问时，对频率做衰减操作，本次访问时间和上次间隔越久，衰减越多，再对频率做增加操作，频率越大增加幅度越小，最大增加至25
- 查看配置

```
config get lfu-decay-time # 控制logc的衰减速度，默认是1，表示xxx分钟衰减一次
config get lfu-log-factor # 控制logc的增长速度，默认是10，值越大，增长越慢
```



## redis锁可能存在什么问题？如何解决？

4个问题

- 锁争抢，【查询锁状态 + 加锁】操作不是原子操作，可能2个节点同时获得锁
- 僵尸锁，持有锁不释放
- 锁过期，锁过期时间设置过短
- 单节点锁，集群宕机不可用

4个解决方案

- 锁争抢，合并【查询锁状态 + 加锁】操作，变成原子操作，防止锁争抢

```
SET key value NX EX 20
```

- 僵尸锁，获得锁时，设置锁过期时间

- 锁过期，设置过期时间时，留好缓冲时间

- 单节点锁，改用redlock，向N个redis节点发送SET NX EX加锁请求；加锁成功数量超过N/2，判定加锁成功；如果加锁失败，所有节点释放锁



redis lua脚本，把多个操作，变成原子操作，例如【校验锁持有人 + 删除锁】



单节点锁的细节

- 键值唯一，记录加锁人身份，删锁时校验
- 设置过期时间，留好缓冲时间
- 加锁失败不等待



# 高可用

## 主从全量复制？

主节点将全量数据同步给从节点，主要包括三个阶段

- 建立连接，协商同步：从节点发送psync命令，携带2个参数（runID = ?, offset = -1），表示要进行数据同步；主节点发送FULLRESYNC命令，表示要进行全量同步，返回2个参数（runID = 123, offset = 0）
- 同步全量数据：主节点执行bgsave命令，生成RDB文件，发送给从服务器；从服务器收到RDB文件，清空当前数据，并读取RDB文件；主节点在此期间（生成、发送、加载RDB），收到的增量写操作命令，记录在replication buffer缓冲区中
- 同步增量写操作：从节点读取RDB文件成功后，发送确认给主节点；主节点将replication buffer缓冲区的增量命令，发送给从节点



## 主从命令传播？

主从节点之间会维护一个长连接，主节点写操作命令，通过长连接发送给从节点

- 命令传播过程：主节点执行命令后，将命令写入repl_backlog_buffer（环形缓冲区，用于找差异数据）缓冲区，再异步发送命令给从节点（听到一件事，先在草稿纸上记录，再告诉别人）
- repl_backlog_buffer用法：主节点通过offset记录【写】位置，从节点通过offset记录【读】位置



## 主节点执行DEL命令，从节点怎么同步？

主节点执行DEL命令后，会将DEL命令发送给从节点，从节点收到命令后，执行DEL命令



## 从节点过多产生什么问题？如何解决？

从节点过多，进行全量同步时，可能会影响主节点，主要有2个问题

- 阻塞主进程：全量同步时，需要通过子进程生成RDB文件，创子进程创建时，需要复制页表，会阻塞主进程
- 占用带宽：传输RDB文件会占用主节点带宽

解决方法

- 将部分从节点升级为经理角色：主节点只需给经理同步数据，经理再给其他从节点同步数据



## 主从增量复制？

主从连接断开又恢复，主从数据采用增量复制方式同步，包含3个步骤

- 步骤1：恢复连接后，从节点发送psync命令，携带2个参数（runID = 123, offset = 100），表示要进行数据同步
- 步骤2：主节点发送CONTINUE命令，表示要进行增量同步
- 步骤3：主节点将增量命令，发送给从节点，如果断线过久，需要全量同步



## 怎么判断redis某个节点是否正常工作？

通过ping-pong心跳机制检测，集群每个节点会定期给其他节点发送ping命令，如果一半以上节点ping一个节点没反应，集群认为这个节点掉线

- 主从模式：主节点每隔10秒ping一次从节点；从节点每隔1秒发送一次replconf ack {offset}命令给主节点，上报当前复制偏移量



## 主从复制，两个buffer有什么区别？

/

- replication buffer：在全量复制阶段使用，记录的是主从RDB传输过程中的增量命令
- repl backlog buffer：在增量复制阶段使用，记录的是主从命令传播过程中的增量命令



## 主从切换，什么数据会丢失？如何解决？

主从切换时，有2部分数据可能会丢失

- 【命令传播】数据丢失：主节点执行命令后，还没来得及将数据同步给从节点，这部分数据会丢失
- 【集群脑裂】数据丢失：网络故障，主节点和客户端连接正常，主节点在集群里失联，触发重新选举，从节点升级为主节点；网络恢复，出现两个主节点（脑裂），旧主节点降级为从节点，旧主节点部分数据会丢失

不能保证数据不丢失，可以减少数据丢失。主要思路是调整【数据同步延迟】阈值，超过阈值后，主节点拒绝写命令，保证旧主节点丢失数据在阈值内，等新主节点上线后，恢复正常

- 减少【命令传播】数据丢失：设置min-slaves-max-lag参数，一旦所有从节点数据同步延迟超过了xxx秒，主节点拒绝写命令

- 减少【集群脑裂】数据丢失：设置min-slaves-to-write参数，规定主节点必须有至少xxx个节点连接，少于阈值，主节点拒绝写命令；设置min-slaves-max-lag参数，一旦所有从节点数据同步延迟超过了xxx秒，主节点拒绝写命令



## 为什么要有哨兵？

哨兵作用于主从模式，作用是实现主从节点故障转移



## 哨兵是如何工作的？

哨兵主要包含三个功能

- 监控：哨兵每秒PING所有主从节点，节点在规定时间内没响应，哨兵标记此节点为【主观下线】，并让其他哨兵投票，如果赞同票超过1/2（达到quoram阈值），标记此节点为【客观下线】，需要在从节点中重新选举主节点
- 选主：



## 哨兵如何判断主节点故障了？

哨兵每秒ping一次节点，如果节点在规定时间内没响应，哨兵会把节点标记为【主观下线】，为了防止误判，会让其他哨兵一起投票，如果赞同票超过1/2（达到quoram阈值），把节点标记为【客观下线】，需要重新选举主节点



## 哪个哨兵进行故障转移？

候选者：第一个发现节点【主观下线】的哨兵，作为leader的候选者，leader执行故障转移

候选者成为leader：让其他哨兵投票，需要同时达成2个条件可以成为leader，【票数超过1/2且达到quoram阈值】，只有候选者可以自己投自己

3个哨兵节点：3个哨兵选举leader只需要2票，2个哨兵选举leader需要2票，挂1个都没法选举成功

quorum最好设置为大于1/2：成功选举客观下线，不一定能成功选举leader进行主从切换

哨兵节点最好设置为奇数：当2个节点同时竞选时，奇数更容易打破平局；相同容错能力下，更节省资源
