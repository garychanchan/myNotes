## 为什么使用redis

redis速度快，可以支撑10w+QPS，是MYSQL的10倍

- redis：华为云单节点4U8G可以支撑10w+QPS，3节点4U8G集群可以支撑30w+QPS
- MYSQL：华为云单节点4U8G可以支撑2w+QPS



## 为什么redis比MYSQL快

redis数据存放在内存中，MYSQL数据存放在磁盘中，内存操作比磁盘操作快

- 数据存储：redis数据存放在内存中；MYSQL数据存放在磁盘中
- 数据结构：redis主要是key-value数据，数据结构简单；MYSQL主要是有表有索引，数据结构复杂



## redis缓存和本地缓存的区别

redis数据存放在（外地服务器）内存中；本地数据存放在（本地服务器）内存中

- 数据存储：redis数据存放在（外地服务器）内存中；本地数据存放在（本地服务器）内存中

- 本地缓存优势：本地数据，不涉及网络传输，访问速度快
- 本地缓存不足：不支持分布式访问；受硬件限制，容量有限

- 如何选择：数据量大，需要支持大规模并发访问，选redis；数据量小，实时性要求高，选本地

## redis锁可能存在什么问题？如何解决？

4个问题

- 锁争抢，【查询锁状态 + 加锁】操作不是原子操作，可能2个节点同时获得锁
- 僵尸锁，持有锁不释放
- 锁过期，锁过期时间设置过短
- 单节点锁，集群宕机不可用

4个解决方案

- 锁争抢，合并【查询锁状态 + 加锁】操作，变成原子操作，防止锁争抢

```
SET key value NX PX 2000
```

- 僵尸锁，获得锁时，设置锁过期时间

- 锁过期，设置锁过期时间，加一个缓冲时间，防止锁提前过期

- 单节点锁，改用redlock，向N个redis节点发送SET NX PX加锁请求；加锁成功数量超过N/2，判定加锁成功；如果加锁失败，所有节点释放锁

分布式锁加锁细节

- 加锁：通过NX合并【查询锁状态 + 加锁】操作，变成原子操作；记录锁持有人信息；设置锁过期时间
- 删除锁：lua脚本合并【校验锁持有人 + 删锁】操作，变成原子操作

```lua
if redis.call("GET",KEYS[1]) == ARGV[1] then
    return redis.call("DEL",KEYS[1])
else
    return -1
end
```



## 什么是大key？大key有什么问题？

key对应的value数据量很大，String类型大小超过100KB，集合类型元素超过1万，属于大key

- 阻塞主线程：大key执行命令慢，长时间占用主线程，导致后续请求等待，甚至请求超时引发雪崩
- 数据倾斜：大key内容存放在同一个分片上，导致分片CPU、内存、带宽升高。内存达到上限，频繁进行内存淘汰；带宽达到上限，请求响应时间升高，连接堆积redis拒绝新连接
- 雪崩：redis服务超时，请求直接访问数据库
- 升华：真实故障，带宽利用率告警



## 怎么发现大key/热key?

大key一般都是悄悄变成大key的，最好的手段是，建立一个持续监控告警的机制

- 云服务监控：云服务监控有大key看板
- prometheus监控：redis服务器安装redis exporter，将数据传输到prometheus



## 如何处理大key?

先评估大key能不能删除或者挪走，不能的话，尝试拆分/压缩大key

- 删除大key：评估大key使用场景，如果没必要用redis作缓存，就删除（unlink或分批次）
- 拆分/压缩大key：将大key拆分成多个小key；如果大key是序列化对象，可以用protobuf代替JSON，压缩数据体积
- 定时清理：对于无限增长的集合，设置定时清理机制



## 什么是热key？热key有什么问题？

key被大量访问，访问频次远高于其他key

- 数据倾斜：热key请求集中在同一分片上，导致分片CPU、带宽升高
- 缓存击穿：热key过期瞬间，大量请求直接访问数据库
- 升华：真实故障，带宽利用率告警



## 如何解决热key

/

- 热点分散：将热key拆分成多个一样的key，每次随机访问其中一个key，分散流量
- 增加本地缓存：先访问本地缓存，查询不到再访问redis（访问redis加锁，同一时间一个请求访问）



## 什么是缓存雪崩？如何解决？

redis宕机或大量redis数据同时过期，导致大量用户请求直接访问数据库

针对大量数据同时过期，主要有3个解决方法

- 避免设置同一过期时间：设置过期时间时，给过期时间加上随机数，避免设置同一过期时间
- 加互斥锁：请求发现redis数据失效后，加互斥锁，保证同一时间只有一个请求构建缓存数据；其他拿不到锁的请求，等待锁释放或返回失败，避免请求继续访问数据库
- 后台更新缓存：后台线程定时检查和更新redis数据，业务线程发现redis数据失效后，不更新redis数据，而是（通过消息队列）通知后台线程更新，避免请求继续访问数据库

redis宕机解决方案

- 组建高可用集群：通过主从节点方式组建高可用集群，主节点宕机，从节点继续提供服务
- 请求限流和熔断：发现redis宕机后，启动限流机制，只允许少量请求访问数据库；或启动熔断机制，暂停redis的访问，直接返回错误，避免请求继续访问数据库



## 什么是缓存击穿？如何解决？

redis某个热点数据过期，导致大量用户请求直接访问数据库，缓存击穿是缓存雪崩的子场景，主要有2个解决方法

- 加互斥锁：请求发现redis数据失效后，加互斥锁，保证同一时间只有一个请求构建缓存数据；其他拿不到锁的请求，等待锁释放或返回失败，避免请求继续访问数据库
- 后台更新缓存：后台线程定时检查和更新redis数据，业务线程发现redis数据失效后，不更新redis数据，而是（通过消息队列）通知后台线程更新，避免请求继续访问数据库



## 什么是缓存穿透？如何解决？

请求访问的数据，既不在redis中，也不在数据库中

- 拒绝非法请求：如果大量非法请求访问不存在的数据，可以在API入口设置入参检查，拒绝非法请求
- 设置默认值：（如果非常紧急没有办法马上改代码发包），可以针对请求访问的数据，在redis中设置一个默认值，避免请求继续访问数据库
- 布隆过滤器：确认数据不在redis后，使用布隆过滤器快速判断数据是否在数据库中，如果不在，直接返回失败，避免请求继续访问数据库
- 布隆过滤器原理：数据写入数据库前，计算数据的哈希值，在位图上标记，查询数据时，如果数据没有被标记，说明数据不在数据库中

- 升华：生产环境发生过缓存穿透的问题，用户通过id获取token，当获取token时，redis会缓存token。用户使用不存在的id获取token，获取失败后立刻重试，导致后端持续获取token，CPU拉到100%。处理故障想了一种方法规避，在redis中给不存在的id设置默认值，获取token失败时，redis也缓存token
