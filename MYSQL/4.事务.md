## 事务特性有哪些？

事务是存储引擎层的实现，有四个特性，ACID

- 原子性（atomicity），事务中的所有操作，要么全部完成，要么全部不完成
- 一致性（atomicity），事务操作前后，数据满足完整性约束（如主键、外键、唯一性等），从一个合法状态转换到另一个合法状态
- 隔离性（isolation），多个事务同时使用相同数据时，保证每个事务都有一个独立的数据空间，对其他事务是隔离的
- 持久性（durability），事务结束后，数据的修改是永久的，系统故障也不会丢失



## ACID如何实现？通过什么技术保证ACID？

/

- 持久性，通过redo log（重做日志）保证
- 原子性，通过undo log（回滚日志）保证
- 隔离性，通过MVCC（多版本并发控制）和锁机制保证
- 一致性，通过持久性+原子性+隔离性保证



## MYSQL可能出现什么并发问题？

可能出现3个并发问题

- 脏读：事务A正在修改数据，事务B读到了事务A【修改过但没提交】的数据
- 不可重复读，一个事务多次读取同一份数据，两次读取到的【数据内容】不一致（侧重数据内容不一致）
- 幻读，一个事务多次读取同一份数据，两次读取到的【数据行数】不一致（侧重数据数量不一致）



## 哪些场景不适合脏读？

/

- 银行：事务A正在修改余额，事务B读取到了事务A【修改过但没提交】的数据，事务B读取到了错误的余额
- 库存：事务A正在修改库存，事务B读取到了事务A【修改过但没提交】的数据，事务B读取到了错误的库存
- 订单，事务A正在修改订单，事务B读取到了事务A【修改过但没提交】的数据，事务B读取到了错误的订单



## 事务隔离级别有哪些？

有4个隔离级别，隔离级别越高，性能越差

- 读未提交，事务能读到其他事务未提交操作
- 读已提交，事务能读到其他事务已提交的操作
- 可重复读，事务重复读取同一份数据，内容是一致的
- 串行化，事务对行数据进行读写操作，如果发生读写冲突（行记录上锁），必须等待前一个事务完成

## 可重复读是怎么避免幻读的？

可重复读通过2个解决方案，很大程度上避免了幻读

- 针对快照读（普通select），通过MVCC避免了幻读。事务启动时创建read view，每次读取同一份快照数据，即使其他事务插入数据，也查询不到这条数据
- 针对当前读（select ... for update），通过加锁避免了幻读（next-key lock记录锁+间隙锁）。执行select ... for update时会给行数据加上next-key lock，其他事务插入数据会被阻塞



## 四种隔离级别如何实现？

/

- 读未提交，读最新的数据
- 读已提交，语句执行前，创建read view
- 可重复读，启动事务时，创建read view，在事务执行过程中，一直用这个read view
- 串行化，加读写锁



## 两种启动事务的时机？

/

- begin/start transaction，执行第一条select语句后，启动事务
- start transaction with consistent snapshot，马上启动事务



## read view四个字段和行数据两个隐藏列的含义及作用？

read view不是数据的快照，是事务id的快照，存储数据量极小

- creator_trx_id，创建read view的事务id
- m_ids，创建read view时，活跃事务id列表（启动了但未提交）
- min_trx_id，创建read view时，最小活跃事务id值
- max_trx_id，创建read view时，下一个新事务的id值（最大事务id值 + 1）
- trx_id，最新修改行数据的事务id。对某条行数据进行改动时，会记录事务id
- roll_pointer，回滚指针，指向数据的历史版本（undo log）。对某条行数据进行改动时，会把旧版本行数据写入undo log，记录undo log指针



## read view在MVCC里是如何工作的？

当事务访问行数据时，会根据隔离级别生成read view，将read review和行数据的trx_id进行比较，可以控制不同事务访问同一个行数据是否可见，称为MVCC（多版本并发控制）

- 行数据trx_id < min，已提交事务，版本对事务可见
- 行数据trx_id >= max，未开始事务，版本对事务不可见
- 行数据trx_id >= min，trx_id不在m_ids列表中，已提交的事务，版本对事务可见
- 行数据trx_id >= min，trx_id在m_ids列表中，已启动但未提交的事务，版本对事务不可见



## 可重复读是如何工作的？

启动事务时，生成read view，在事务执行过程中，一直用这个read view

![image-20260312234129169](C:\Users\heyga\AppData\Roaming\Typora\typora-user-images\image-20260312234129169.png)

- 事务B读取余额，行数据trx_id为50，trx_id < min，表示已提交的事务，该版本对事务可见，读到余额100万
- 事务A修改余额，新增一条trx_id为51的行数据，undo_pointer指向trx_id为50的行数据，形成版本链
- 事务B读取余额，行数据trx_id为51，在m_ids列表中，表示活跃的事务，该版本对事务不可见，顺着undo_pointer版本链找旧版本数据，直到找到可见的数据，行数据trx_id为50，读取到余额100万
- 事务A提交事务
- 事务B读取余额，行数据trx_id为51，在m_ids列表中，表示活跃的事务，该版本对事务不可见，顺着undo_pointer版本链找旧版本数据，直到找到可见的数据，行数据trx_id为50，读取到余额100万
